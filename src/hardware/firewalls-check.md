# Посібник з перевірки роботи фаєрвола (iptables/nftables)

Перевірка захисту — це дуже складний процес, тому що 1) неможливо передбачити всі варіанти атак, які можуть провести зловмисники, 2) звʼязок чудово працює коли захист *не* працює, тому прогалини у захисті не помітні при звичайній роботі.

## 1. Перевірка поточних правил

Перший крок — переконатися, що правила взагалі є і увімкнені. Для цього потрібно перевірити конфігурацію фаєрволу. Для роутерів Mikrotik, правила можна подивитися у IP -> Firewal -> Filter rules (IP -> Фаєрвол -> Правила фільтрування).

Правила для захисту локальної мережі знаходяться у ланцюжку FORWARD (пересилання). Правила для захисту самого роутера — у ланцюжку INPUT (вхід). Правила для обмеження роботи, щоб наприклад взломаний роутер не змі

Якщо правила увімкнені та працюють, то лічильники оброблених пакетів біля них будуть ненульові через те, що зараз в Інтернет завжди є якийсь шкідливий чи помилковий трафік.

## 2. Тестування проходження трафіку (Traffic Testing)

Щоб зрозуміти, чи працює правило `DROP` або `ACCEPT`, потрібно спробувати ініціювати з'єднання з іншої машини.

Як правило, роутер підʼєднаний до інтернету через один порт, і метод прямого тестування — непрактичний, тому що роутер треба відключати від інтернету та підключати до тесувальної машини, що припиняє звʼязок.

Якщо є запасний роутер, сконфігурований ідентично до основного роутера, то можна використати його.

Якщо роутер сконфігурований так що може зʼєднуватися з Інтернетом через два порти одночасно, то можна тестувальну машину підключити до запасного порту.

Якщо фаєрвол увімкнений на окремому пристрої, то можна його зняти на час тестування — захист послабиться, але звʼязок працюватиме.

### Використання `nmap` (Сканування портів)

Для перевірки роутера, чи відкриті порти та чи працює фільтрація, можна використати nmap (або його графічну версію: zenmap).

```bash
# Сканування конкретного порту (наприклад, 80)
nmap -p 80 <зовнішня_IP_адреса_роутера>

# Сканування на наявність будь-яких відкритих портів
nmap <зовнішня_IP_адреса_роутера>
```
* Якщо порт `filtered` (відфільтровано) — пакет відхилено фаєрволом.
* Якщо `closed` (закрито) — пакет пройшов, але на порту немає служби.
* Якщо `open` (відкрито) — працює правило `ACCEPT` і порт відкритий.

Часто, на роутері можуть бути відкриті порти для віддаленого конфігурування роутера ззовні, такі як 23 (ssh) чи 80 (http) чи 443 (https), або інші. Для основного роутера їх потрібно закрити або через фаєрвол, або вимкнувши відповідні сервіси, щоб вони не слухали запити ззовні.

Також, часто буває що клієнтські машини динамічно відкривають якісь порти на роутері через сервіс Plug&Play (Увімкни і Працюй), наприклад для телефонії, чи відеозвʼязку, чи для віддаленого доступу до робочого місця. Без прямого доступу, такі програми будуть змушені використовувати сервіс-посередник, що погіршить якість звʼязку. Якщо вимкнути підтримку Plug&Play, то безпека покращиться але якість звʼязку погіршиться.

### Зміна адреси на адресу заблокованого ресурсу

Можна підключити роутер до машини і встановити на машині адресу забороненого ресурсу, після чого можна буде спробувати перевірити, чи дійсно працює заборона.

## Чек-лист перед завершенням робіт
1. [ ] Перевіряйти чи правила `DROP` не заблокували ваш власний доступ (завжди перевіряйте `INPUT`!).
2. [ ] Лічильники пакетів збільшуються при активності.
3. [ ] Правила `NAT` не конфліктують з правилами `FILTER`.
4.  [ ] Правила не блокують легітимний трафік.
